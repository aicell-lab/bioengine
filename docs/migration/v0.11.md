# Migrating BioEngine apps to v0.11 (manifest format 0.6.0)

BioEngine v0.11 made a hard break in how apps are authored. There is no
backwards-compatibility shim: manifests using the v0.5 layout (with a
`deployments:` list and no `format_version: 0.6.0`) are rejected with a
clear error pointing here.

This document describes the new layout and walks through migrating an
existing app.

## What changed and why

The v0.5 model downloaded each `.py` file from your Hypha artifact, ran it
through `exec()` inside the worker process, and monkey-patched the user
class to wire up lifecycle hooks. Two consequences:

1. **No top-of-file imports of non-worker deps.** Any heavy import
   (`pandas`, `torch`, `numpy`, ...) had to live inside method bodies.
2. **No relative or cross-file imports.** Each file ran in an isolated
   globals dict, so a shared `utils.py` was impossible.

The v0.11 model ships your app's Python *package* as a Ray
`runtime_env.py_modules` upload. The worker never imports your code; it
runs an introspection Ray task in the app's runtime env that walks the
type-hint composition graph and returns a JSON spec. Both constraints
above disappear.

## The new layout

```
my-app/                       ← artifact root + Python module directory
├── manifest.yaml             ← v0.6.0 format
├── README.md
├── tutorial.ipynb
├── frontend/index.html
└── deployment.py             ← @bioengine.app class
```

The artifact root *is* the Python module directory. Author-facing
content (`manifest.yaml`, `README*`, `*.md`, `*.ipynb`, `frontend/`,
images) stays in the artifact for Hypha to serve but is excluded from
the `py_modules` upload that ships to Ray Serve replicas.

For multi-file apps (composition, shared helpers), keep adding files
at the artifact root and use plain top-level imports:

```
composition-demo/
├── manifest.yaml
├── frontend/index.html
├── entry.py                  ← @bioengine.app entry — type-hints reference RuntimeA/B/C
├── utils.py
└── runtimes/
    ├── __init__.py
    ├── a.py
    ├── b.py
    └── c.py
```

`entry.py` writes `from utils import shout` and `from runtimes.a import RuntimeA`.

## The new manifest

```yaml
name: My App
id: my-app
id_emoji: '🔬'
description: ...
type: ray-serve                              # kept
format_version: 0.6.0                        # required gate
version: 1.0.0
entry: deployment:MyApp                      # replaces deployments: [file:Class]
frontend_entry: frontend/index.html          # optional
authorized_users: ['*']                      # optional
```

Notes:

* `deployments:` is removed. Multi-deployment composition is now declared
  in code via type hints (see below).
* `entry:` is `module:ClassName` where `module` is the filename (without
  `.py`) of any module at the artifact root. Dotted paths
  (`subpkg.module:ClassName`) are accepted for files inside a
  sub-package.
* `type:` stays `ray-serve`.

## The new authoring API

```python
import bioengine
from pydantic import Field

@bioengine.app(num_cpus=1, memory_mb=512, pip=["pandas"])
class MyApp:
    def __init__(self, threshold: float = 0.5):
        self.threshold = threshold

    @bioengine.async_init
    async def load(self):
        """Optional one-shot async init. Name is free."""
        self.model = await load_my_model()

    @bioengine.smoke_test
    async def _smoke(self):
        """Optional startup smoke test. Raise to fail the replica."""
        result = await self.predict(x=[1.0])
        assert result["ok"]

    @bioengine.health_check
    async def _health(self):
        """Optional periodic health check."""
        ...

    @bioengine.method
    async def predict(self, x: list = Field(..., description="input")) -> dict:
        """Exposed via Hypha RPC."""
        return {"ok": True, "y": [v + self.threshold for v in x]}

    @bioengine.multiplexed(max_models=3)
    async def _get_model(self, model_id: str):
        """Multiplexed model loader."""
        return load_one(model_id)
```

## Composition by type hint

The legacy model required parameter names matching filenames in the
manifest. The new model derives the graph from type hints:

```python
# runtimes/a.py
import bioengine

@bioengine.app(num_cpus=1)
class RuntimeA:
    @bioengine.method
    async def ping(self) -> str:
        return "pong"


# entry.py
import bioengine
from runtimes.a import RuntimeA

@bioengine.app(num_cpus=0)
class Entry:
    def __init__(self, runtime_a: RuntimeA):     # type hint drives composition
        self.runtime_a = runtime_a

    @bioengine.method
    async def ping(self) -> str:
        return await self.runtime_a.ping()       # no .remote() — proxy hides it
```

`.remote()` is hidden by the `BioEngineRuntimeHandle` proxy. The raw
Ray Serve handle is reachable via `self.runtime_a._raw` if you need
streaming, batching options, or other Ray-specific behaviour.

## The decorator-module import rule

Modules that contain `@bioengine.app` — and anything they transitively
import at top level — must be importable with just `bioengine[worker]`
and the standard library available. The worker's introspection Ray task
walks the type-hint composition graph by actually importing those
modules in a clean runtime_env that has only the BioEngine baseline
installed.

Heavy application dependencies still ship to the replica via the
decorator's `pip=` arg. They just can't be top-of-file imports inside
a decorator-bearing module.

The recommended patterns:

**Inside-method import** — simplest, no extra files:

```python
import bioengine
from pydantic import Field

@bioengine.app(num_cpus=1, pip=["pandas"])
class MyApp:
    @bioengine.method
    async def analyse(self, rows: list = Field(...)) -> dict:
        import pandas as pd                # imported when the method runs
        df = pd.DataFrame(rows)
        return df.describe().to_dict()
```

**Sibling helper module** — heavier work goes in a non-decorator
module, lazy-imported from the decorator file:

```python
# numpy_ops.py — NOT a decorator file, not imported at top level from one
import numpy as np

def stats(values: list) -> dict:
    arr = np.array(values, dtype=float)
    return {"mean": float(np.mean(arr)), "std": float(np.std(arr))}
```

```python
# runtimes/b.py — @bioengine.app host, light at top level
import bioengine

@bioengine.app(num_cpus=1, pip=["numpy==1.26.4"])
class RuntimeB:
    @bioengine.method
    async def analyse(self, values: list) -> dict:
        from numpy_ops import stats        # lazy
        return stats(values)
```

If the introspection task fails with `ImportError`, the message names
the missing module — move that import inside a method or factor it
into a sibling helper.

## Module-level accessors

`bioengine.datasets` is the process-local `BioEngineDatasets` singleton;
`bioengine.logger` is the Ray Serve logger. Both replace the
`self.bioengine_datasets` instance attribute injected by the v0.5
framework.

```python
import bioengine

@bioengine.app(num_cpus=1)
class MyApp:
    @bioengine.method
    async def list_files(self) -> dict:
        return await bioengine.datasets.list_datasets()
```

Touching `bioengine.datasets` at module import time will raise
`MissingDataServerError` (the introspection task does not configure a
data server URL). Always touch it from inside instance methods.

## Reserved method names

Methods literally named `check_health`, `async_init`, or `test_deployment`
are now rejected at decoration time. Use the corresponding decorator
(`@bioengine.health_check`, `@bioengine.async_init`,
`@bioengine.smoke_test`) on a method with any name you like.

## Decorator → decorator mapping

| Old | New |
|---|---|
| `@serve.deployment(ray_actor_options={...})` | `@bioengine.app(num_cpus=, num_gpus=, memory_mb=, pip=[], env_vars={}, max_ongoing_requests=)` |
| `@schema_method` | `@bioengine.method` |
| `async def async_init(self): ...` (magic name) | `@bioengine.async_init\nasync def whatever(self): ...` |
| `async def test_deployment(self): ...` | `@bioengine.smoke_test\nasync def whatever(self): ...` |
| `async def check_health(self): ...` | `@bioengine.health_check\nasync def whatever(self): ...` |
| `@serve.multiplexed(max_num_models_per_replica=3)` | `@bioengine.multiplexed(max_models=3)` |
| `self.bioengine_datasets.X(...)` | `bioengine.datasets.X(...)` |
| `await self.runtime_a.ping.remote()` | `await self.runtime_a.ping()` |

`@serve.deployment` and `@serve.multiplexed` still work if you need
escape hatches — `@bioengine.app` itself wraps `@serve.deployment` under
the hood. You can also pass `ray_actor_options={...}` to `@bioengine.app`
for fields not exposed as flat kwargs; the dict is deep-merged on top of
the flat args.

## Step-by-step migration

For each app:

1. **Keep the `.py` files at the artifact root.** No nested Python
   package directory. Sub-packages (`runtimes/`, `models/`, ...) are
   fine when they help group related modules.

2. **Update `manifest.yaml`.** Remove `deployments:`, add
   `format_version: 0.6.0` and `entry: module:ClassName` (module is the
   filename of the entry file, without `.py`).

3. **Update each `.py` file.**
   * Replace `@serve.deployment(ray_actor_options={...})` with
     `@bioengine.app(num_cpus=..., memory_mb=..., pip=[...])`.
   * Replace `@schema_method` with `@bioengine.method`.
   * Wrap lifecycle methods with their decorators and rename if you wish.
   * Replace `self.bioengine_datasets` with `bioengine.datasets`.
   * If you have multi-deployment composition, replace `DeploymentHandle`
     parameter annotations with the actual class types and drop
     `.remote()` from call sites.

4. **Validate locally** before deploying:
   ```
   bioengine apps validate ./my-app
   ```

5. **Upload and deploy** as before:
   ```
   bioengine apps upload ./my-app
   bioengine apps run <artifact-id>
   ```

## Reference migrations

Two reference apps in this repo have been migrated and serve as
end-to-end examples:

* `apps/demo-app/` — single-file app with every lifecycle hook and
  module-level dataset/logger access.
* `apps/composition-demo/` — multi-file app with shared `utils.py`,
  a `runtimes/` sub-package, type-hint composition, and a top-of-file
  `import numpy as np`.
